# Multi-Pass Audit — 2026-05-08

A focused review across the contracts in `solidity/contracts/`. Done in
multiple passes by one model in one session — not a substitute for a real
audit, but an honest attempt to find issues before mainnet.

Scope: CawActions, CawProfile, CawProfileL2, CawActionsArchive,
CawChallengeRelay, CawClientManager, CawProfileMinter,
CawProfileMarketplace, CawBuyAndBurn, plus auxiliaries (PathwayExpander,
OnlyOnce, Quoter, etc.).

Severity legend:
- 🔴 Critical / High — exploitable or loss-of-funds
- 🟠 Medium — correctness bug, design gap, or DoS surface
- 🟡 Low — footgun, surprising behaviour
- 🟢 Informational — code quality / efficiency
- ⚪ Verified safe (noted to confirm I looked, not a finding)

## Findings fixed in this audit pass

### 🔴 ZK-B: ZK path didn't verify session expiry → expired keys still authorized actions
- **File**: `CawActions.sol`, `_zkProcessOneGroup`
- **Bug**: The session-key branch loaded scope/spendLimit/tipRate from
  `cawProfile.sessions(...)` but skipped the expiry check. An expired or
  revoked-by-time session key would still authorize actions in the ZK
  path even though `processActions` (sig path) rejected them.
- **Fix**: Added `require(expiry > block.timestamp, "Session expired or
  not found")` after the session lookup. Same error string the sig path
  uses, so callers see consistent errors.
- **Regression test**: `test/zk-actions-test.js` — "regression: rejects
  expired session key in ZK path". Verified by stash-revert-rerun: with
  the fix reverted the test fails ("expected revert when session is
  expired"), proving the bug existed.
- **Commit**: `436d612`

### 🔴 ZK-C documented: ZK path explicitly rejects ERC-1271 / contract-owned profiles
- **File**: `CawActions.sol`, `_zkProcessOneGroup`
- **Status**: Not a bug — a design gap. The proof commits to the
  EOA-recovered signer; for a Safe-owned profile there is no EOA. The
  contract now documents that the non-owner branch requires a valid
  session record OR reverts (no 1271 fallback by design). Validators
  must route Safe-owned profiles through `processActions` (which DOES
  have a 1271 fallback). Caller responsibility documented in the
  natspec.
- **Commit**: `436d612`

### 🔴 L2-2: registerSessionPersonal replay-vulnerable
- **File**: `CawProfileL2.sol`
- **Bug**: The personal-sign message has a fixed shape (13 lines, no
  nonce). The function bumped `sessionNonce[signer]` after recovery,
  but that bump only invalidated pending EIP-712 `registerSession`
  messages — it did NOT prevent the same personal-sign payload from
  being re-submitted to undo a revocation.
- **Attack**: User signs personal-sign authorising sessionKey X with
  expiry T. Anyone submits, session created. User revokes X. Attacker
  re-submits the SAME signed payload (anytime before T) → session
  re-created without the user's knowledge.
- **Fix**: Track `consumedSessionMessage[bytes32]` keyed by digest;
  first submission marks consumed, replays revert with "Message already
  consumed".
- **Regression test**: `test/session-personal-replay-test.js`. Verified
  by revert-restore.
- **Commit**: `d9a02ac`

### 🟡 L2-5: bypassLZ setWithdrawable trapped native ETH
- **File**: `CawProfileL2.sol`
- **Bug**: setWithdrawable is `external payable`. In bypassLZ
  co-deployment mode no LayerZero send happens, but a misconfigured
  validator forwarding non-zero `msg.value` from a stale withdrawQuote
  would have ETH trapped in the contract (no sweep path).
- **Fix**: `require(msg.value == 0)` in the bypassLZ branch. Validators
  computing the fee correctly are unaffected; misconfigured ones now
  revert immediately instead of bricking funds.
- **Commit**: `d9a02ac`

### 🟠 ARC-3: validatorSubmissions array unbounded → slash DoS
- **File**: `CawActionsArchive.sol`
- **Bug**: `validatorSubmissions[validator]` accumulated every
  submission ever made, including finalized ones. The slash loops in
  `resolveChallenge` and `slashIncoherentRoot` iterated this array. With
  ~600+ historical submissions (in a long-tenured validator's history),
  the slash tx approached Base mainnet's 30M-gas-per-block ceiling and
  could OOG, blocking legitimate fraud resolution.
- **Fix**: Prune via swap-and-pop on `finalizeSubmission`. The array
  now contains only PENDING submissions; finalized ones are removed.
  Slash loop length is bounded by `pendingCount[validator]`.
- **Commit**: `45ef275`

## Verified-safe items I looked at carefully

### CawActions
- Replay across contracts/chains (eip712DomainHash binds proof + sig)
- Per-group signer-uniformity check (prevents prover smuggling
  unauthorized signer onto batched action)
- No path executes the same action twice (cawonce burn before
  processing the rest)
- `clientHash` only extends on executed actions
- Session keys can't escalate (`!ba.isSessionKey` gate on
  `_handleSessionAction`)
- ZK path's `signers[]` is bound positionally to actions via the
  circuit's commit ordering
- Solidity reads of `signers[]` byte-match what the proof commits to
- ECDSA recovery to address(0) is rejected by the SP1 circuit (panic)
- No reentrancy surface in processActionsWithZkSigs (all cross-contract
  calls are view OR L2 mutations that don't loop back)

### CawProfile (L1)
- `_withdrawFees` correctly CEI-ordered
- ETH math in fee accounting is consistent (2× per fee event, drained
  symmetrically by `_withdrawFees`)
- `mainnetLzId` is excluded from `peerIds`, so
  `peerWithMaxPendingTransfers()` never returns it →
  `_updateNewOwners(mainnetLzId, ...)` is dead code (would also
  fail-closed if reached, since L2 `updateOwners` requires `fromLZ`)
- 7 LZ-whitelisted selectors are all unique and don't collide with
  inherited Ownable/ERC721/OApp functions

### CawProfileL2
- 7 LZ-whitelisted selectors are unique
- `addToBalance` triple-gate: `fromLZ || cawActions || (bypassLZ &&
  cawProfile)` — all three paths verified
- Session nonce-bump paths: every register flow bumps; revoke doesn't
  need to (registerSession bumps after success, blocking replay)
- `revokeSessionBySig` replay is harmless (idempotent + precondition)
- ownerOf returns address(0) for non-existent tokens by design (cited
  in storage layout); session lookup against address(0) is empty record
  → expiry=0 → fails the new expiry check

### CawActionsArchive
- `nonReentrant` on `withdraw`, `resolveChallenge`,
  `slashIncoherentRoot` blocks re-entry into the slash flow
- `_lzReceive` try/catch self-call prevents channel stalls from
  malformed messages
- `submitReplication` validates checkpoint range, action count, r
  length, no double-claim
- `slashIncoherentRoot`'s data-commitment check (line 446) ensures
  caller is re-supplying the exact bytes the submitter committed to
- `_buildMerkleRoot` matches the off-chain `merkletreejs` implementation
  (sortPairs + leaf double-hash)

### CawProfileMarketplace
- `nonReentrant` on every external entry
- `cancelListing` for English with active bid uses pull-pattern refund
  (prevents bidder-contract DoS blocking cancel)
- `placeBid` push-then-fallback-to-pull is correct CEI; 2300-gas stipend
  is enough for EOA receivers, smart wallets fall back to pendingReturns
- `withdrawBid` zeroes pendingReturns BEFORE the call (CEI)
- `reclaimBid` is the safety valve for "seller transferred NFT mid-auction"
- Reentrancy via `seller.call{value: price}` in `buy` is contained:
  re-listing during reentry creates a stale listing that nobody can buy
  (subsequent buy attempts revert because seller no longer owns NFT)

### CawProfileMinter
- Username availability is checked atomically (no front-run window
  WITHIN a single tx; cross-tx race resolves to the first miner)
- Burn happens before NFT mint; revert in mint path rolls back burn
- Self-mint-only constraint on bundled QuickSign flows prevents third
  parties from injecting sessions into other wallets

### CawChallengeRelay
- Permissionless by design; anyone can challenge
- `_lzReceive` reverts (relay doesn't receive) — channel stays alive
  because nobody peers back at the relay

### PathwayExpander
- Owner can only ADD peers for new eids; never modify existing
- Defense-in-depth `peers[eid] == 0` check before delegating to setPeer

## Known-but-unfixed items (informational / by-design)

### 🟡 CawProfile fallback() accepts ETH silently
`fallback() external payable {}` exists with empty body. Random calls
(non-matching selectors) succeed and absorb msg.value. Removing it
would save ~50 bytes and make accidental sends revert. Left as-is to
avoid breaking any LZ V2 internal that may rely on it; the worst
outcome is users losing ETH to typos. Recoverable: no.

### 🟡 SAFE-5 firstClientId can misreport in safeProcessActions
If the first sig group fails, `sc.firstClientId` reflects the failed
group's clientId. Subsequent successful groups may have different
clientIds. The `ActionsProcessed` event reports the (failed-group)
firstClientId. Indexer hint, not a security issue — per-action
clientIds in the packedActions blob are the source of truth.

### 🟢 CawActions.externalSelf is unused
Set in constructor (line 161) but never read. Removing would save ~50
bytes. Low priority.

### 🟢 CawActions has 31 unique reason strings (~1KB bytecode)
Migrating to custom errors (`error Foo(); revert Foo();`) would save
300-700 bytes. Not safety-critical; mostly a size optimization for the
EIP-170-cap-adjacent contracts.

## What I did NOT cover (caveats)

- No fuzz testing
- No symbolic execution
- No formal-verification properties
- No exhaustive cross-contract reentrancy enumeration (looked at the
  obvious paths only)
- The CawProfileURI / CawFontDataA / CawFontDataB SVG-rendering contracts
  were not reviewed (low risk surface for security; they're view-only)
- LayerZero V2 OApp internals were trusted (treated as audited dependency)
- Uniswap V2 / SwapRouter integration was treated as audited dependency
- The SP1 zkVM and Groth16 verifier soundness was trusted (NCC + Cantina
  audited)

**Recommendation**: a real third-party audit before mainnet. The fixes
landed in this pass close real bugs but the surface is large; finding
two HIGH-severity bugs in one focused pass suggests there are more I
missed.
