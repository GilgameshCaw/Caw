# Multi-Pass Audit — 2026-05-08

A focused review across the contracts in `solidity/contracts/`. Done in
multiple passes by one model in one session, then extended with a second
session (Round 2 below) that ran 6 parallel background audits — one per
contract cluster — plus my own 6 sequential rounds focused on
cross-contract interactions, storage layout, numerical edges,
adversarial users, LayerZero failure modes, and sig/ZK path divergence.

This is an honest attempt to find issues before mainnet. Round 2 found
9 additional issues including 3 HIGH-severity bugs that the first pass
missed.

Scope: CawActions, CawProfile, CawProfileL2, CawActionsArchive,
CawChallengeRelay, CawClientManager, CawProfileMinter,
CawProfileMarketplace, CawBuyAndBurn, plus auxiliaries (PathwayExpander,
OnlyOnce, Quoter, etc.).

Severity legend:
- 🔴 Critical / High — exploitable or loss-of-funds
- 🟠 Medium — correctness bug, design gap, or DoS surface
- 🟡 Low — footgun, surprising behaviour
- 🟢 Informational — code quality / efficiency
- ⚪ Verified safe (noted to confirm I looked, not a finding)

## Findings fixed in this audit pass

### 🔴 ZK-B: ZK path didn't verify session expiry → expired keys still authorized actions
- **File**: `CawActions.sol`, `_zkProcessOneGroup`
- **Bug**: The session-key branch loaded scope/spendLimit/tipRate from
  `cawProfile.sessions(...)` but skipped the expiry check. An expired or
  revoked-by-time session key would still authorize actions in the ZK
  path even though `processActions` (sig path) rejected them.
- **Fix**: Added `require(expiry > block.timestamp, "Session expired or
  not found")` after the session lookup. Same error string the sig path
  uses, so callers see consistent errors.
- **Regression test**: `test/zk-actions-test.js` — "regression: rejects
  expired session key in ZK path". Verified by stash-revert-rerun: with
  the fix reverted the test fails ("expected revert when session is
  expired"), proving the bug existed.
- **Commit**: `436d612`

### 🔴 ZK-C documented: ZK path explicitly rejects ERC-1271 / contract-owned profiles
- **File**: `CawActions.sol`, `_zkProcessOneGroup`
- **Status**: Not a bug — a design gap. The proof commits to the
  EOA-recovered signer; for a Safe-owned profile there is no EOA. The
  contract now documents that the non-owner branch requires a valid
  session record OR reverts (no 1271 fallback by design). Validators
  must route Safe-owned profiles through `processActions` (which DOES
  have a 1271 fallback). Caller responsibility documented in the
  natspec.
- **Commit**: `436d612`

### 🔴 L2-2: registerSessionPersonal replay-vulnerable
- **File**: `CawProfileL2.sol`
- **Bug**: The personal-sign message has a fixed shape (13 lines, no
  nonce). The function bumped `sessionNonce[signer]` after recovery,
  but that bump only invalidated pending EIP-712 `registerSession`
  messages — it did NOT prevent the same personal-sign payload from
  being re-submitted to undo a revocation.
- **Attack**: User signs personal-sign authorising sessionKey X with
  expiry T. Anyone submits, session created. User revokes X. Attacker
  re-submits the SAME signed payload (anytime before T) → session
  re-created without the user's knowledge.
- **Fix**: Track `consumedSessionMessage[bytes32]` keyed by digest;
  first submission marks consumed, replays revert with "Message already
  consumed".
- **Regression test**: `test/session-personal-replay-test.js`. Verified
  by revert-restore.
- **Commit**: `d9a02ac`

### 🟡 L2-5: bypassLZ setWithdrawable trapped native ETH
- **File**: `CawProfileL2.sol`
- **Bug**: setWithdrawable is `external payable`. In bypassLZ
  co-deployment mode no LayerZero send happens, but a misconfigured
  validator forwarding non-zero `msg.value` from a stale withdrawQuote
  would have ETH trapped in the contract (no sweep path).
- **Fix**: `require(msg.value == 0)` in the bypassLZ branch. Validators
  computing the fee correctly are unaffected; misconfigured ones now
  revert immediately instead of bricking funds.
- **Commit**: `d9a02ac`

### 🟠 ARC-3: validatorSubmissions array unbounded → slash DoS
- **File**: `CawActionsArchive.sol`
- **Bug**: `validatorSubmissions[validator]` accumulated every
  submission ever made, including finalized ones. The slash loops in
  `resolveChallenge` and `slashIncoherentRoot` iterated this array. With
  ~600+ historical submissions (in a long-tenured validator's history),
  the slash tx approached Base mainnet's 30M-gas-per-block ceiling and
  could OOG, blocking legitimate fraud resolution.
- **Fix**: Prune via swap-and-pop on `finalizeSubmission`. The array
  now contains only PENDING submissions; finalized ones are removed.
  Slash loop length is bounded by `pendingCount[validator]`.
- **Commit**: `45ef275`

- `_lzReceive` reverts (relay doesn't receive) — channel stays alive

### PathwayExpander
- Owner can only ADD peers for new eids; never modify existing
- Defense-in-depth `peers[eid] == 0` check before delegating to setPeer

## Known-but-unfixed items (informational / by-design)

### 🟡 CawProfile fallback() accepts ETH silently
`fallback() external payable {}` exists with empty body. Random calls
(non-matching selectors) succeed and absorb msg.value. Removing it
would save ~50 bytes and make accidental sends revert. Left as-is to
avoid breaking any LZ V2 internal that may rely on it; the worst
outcome is users losing ETH to typos. Recoverable: no.

### 🟡 SAFE-5 firstClientId can misreport in safeProcessActions
If the first sig group fails, `sc.firstClientId` reflects the failed
group's clientId. Subsequent successful groups may have different
clientIds. The `ActionsProcessed` event reports the (failed-group)
firstClientId. Indexer hint, not a security issue — per-action
clientIds in the packedActions blob are the source of truth.

### 🟢 CawActions.externalSelf is unused
Set in constructor (line 161) but never read. Removing would save ~50
bytes. Low priority.

### 🟢 CawActions has 31 unique reason strings (~1KB bytecode)
Migrating to custom errors (`error Foo(); revert Foo();`) would save
300-700 bytes. Not safety-critical; mostly a size optimization for the
EIP-170-cap-adjacent contracts.

# Round 2 — Parallel multi-agent audit + sequential cross-cutting passes

A second session ran 6 fresh-eyes audit agents in parallel (one per contract
cluster) plus 6 sequential cross-cutting rounds by the main reviewer. This
section tracks the additional findings.

## Round 2 findings fixed

### 🔴 C-1 (HIGH): WITHDRAW silently lost CAW in bypassLZ co-deployment mode
- **Source**: CawProfileL2 audit agent
- **Files**: `CawProfile.sol::setWithdrawable`, `CawActions.sol` × 3 entry points
- **Bug**: `_executeWithdrawals` was gated on `withdrawFee > 0`. In bypassLZ
  mode (mainnet co-deployment) the fee is legitimately 0 (no LZ message
  needed), so the call was skipped entirely and L1's `withdrawable[]` was
  never credited. The L2 side had ALREADY debited via `withdrawTokens()` in
  `_applyAction`, so the user permanently lost CAW.
- **Attack**: Any WITHDRAW submitted in bypassLZ mode loses funds. Dormant
  today (no client uses bypassLZ), but fires the moment one does.
- **Fix**: gate on `withdrawFee > 0 || cawProfile.bypassLZ()` across all
  three entry points. Also added a non-LZ branch to L1's `setWithdrawable`
  for direct calls from the co-deployed L2 mirror.
- **Commit**: `7e3b2e4`

### 🔴 H-1 / WITHDRAW (HIGH): WITHDRAW with `recipients=[]` double-debits user
- **Source**: CawActions audit agent
- **File**: `CawActions.sol::_distributeAmountsMem`
- **Bug**: For a WITHDRAW with `recipients=[]` and `amounts=[X]`, the
  `hasExplicitTip = numAmounts == numRecipients + 1` condition matches
  (`1 == 0+1`). The function then BOTH (a) re-spends X via
  `spendAndDistribute` AND (b) credits X to the validator as a "tip". The
  user pays 2X, the validator pockets X. The L1 withdraw of X also fires
  in `_applyAction`. Net: user pays 2X for a 1X withdraw.
- **Attack**: A malicious FE/SDK convinces a user to sign WITHDRAW with
  empty recipients (the simplest natural shape). The user-signed message
  appears identical to any other WITHDRAW; on-chain the validator pockets
  X CAW for free.
- **Fix**: explicit
  `require(!(isWithdrawal && numRecipients == 0 && hasExplicitTip))` —
  rejects the ambiguous shape. Canonical shape is `recipients=[senderId]`
  (used by all existing tests and the FE).
- **Commit**: `7e3b2e4`

### 🔴 H-1 / authenticate (HIGH): `authenticate()` doesn't subscribe chain → impersonation after transfer
- **Source**: CawProfile L1 audit agent
- **File**: `CawProfile.sol::authenticate`
- **Bug**: `mintAndAuth` and `mintAndDeposit` do `chosenChainIds.add()` but
  the standalone `authenticate()` doesn't. Tokens authenticated to a chain
  via `authenticate()` (rather than at mint time) are NOT registered for
  cross-chain ownership-sync. After the L1 owner transfers the NFT, the
  new owner is never propagated to that chain. The OLD owner can keep
  posting as the username on that chain indefinitely.
- **Fix**: add `chosenChainIds[tokenId].add(uint256(lzDestId))` at the top
  of `authenticate()`, mirroring the other entry points.
- **Commit**: `7e3b2e4`

### 🔴 M-1 (HIGH): Expired session falls through to ERC-1271 → owner authority
- **Source**: CawActions audit agent
- **File**: `CawActions.sol::_verifySignatureMem`, `_verifyBatchSignature`
- **Bug**: Control flow checked owner-match, then session-record-active,
  then fell into ERC-1271 fallback. For a Safe-owned profile where the
  expired session signer was ALSO a Safe-validated key, the fallback
  silently accepted as `(owner, false)` — granting full owner authority
  instead of rejecting the expired session.
- **Attack**: User registers Safe-owner key K as a session with low
  spendLimit and short expiry (their own safety boundary). Session
  expires. Anyone with K signs CAW actions, gets owner authority via the
  1271 fallback.
- **Fix**: explicit `if (expiry != 0) revert("Session expired")` after the
  active-session check, before the 1271 fallback. Applied to both
  signature verification functions.
- **Commit**: `2808d8c`

### 🟠 M-2 (MEDIUM): `depositFor` permissionless + zero-amount → chosenChainIds pollution
- **Source**: CawProfile L1 audit agent
- **File**: `CawProfile.sol::depositFor`
- **Bug**: `depositFor` is intentionally permissionless (routers
  pay-on-behalf-of). With `amount=0`, a third party can permanently
  subscribe a victim's token to chains the victim never opted into, and
  force-authenticate to clients the victim didn't pick. The victim then
  pays per-transfer LZ fees for chains they don't use.
- **Fix**: `require(amount > 0)` in `depositFor`. Routers always have
  non-zero deposits; the zero-amount path was a grief vector.
- **Commit**: `7e3b2e4`

### 🟠 M-3 (MEDIUM): `peerWithMaxPendingTransfers` panics on empty `peerIds`
- **Source**: CawProfile L1 audit agent
- **File**: `CawProfile.sol::peerWithMaxPendingTransfers`, `_updateNewOwners`
- **Bug**: `peerIds.at(0)` reverts with panic on an empty set. In a
  single-chain mainnet bypassLZ deployment, `peerIds.length() == 0`,
  bricking `mint()`, `withdrawTo()`, and `transferAndSync()`.
- **Fix**: early-return sentinel 0 from the function; treat 0 as no-op in
  `_updateNewOwners`.
- **Commit**: `7e3b2e4`

### 🟠 M-3 / qs (MEDIUM): Malformed qs:/qx: payload reverts whole batch
- **Source**: CawActions audit agent
- **File**: `CawActions.sol::_handleSessionAction`
- **Bug**: `require(t.length == 71)` for qs: and `require(t.length == 23)`
  for qx: meant one malicious user's malformed OTHER action would tank
  the entire batch (sig path all-or-nothing; ZK path also reverts whole
  group rather than skipping).
- **Fix**: replace `require()` with silent `return`. Malformed payloads
  no-op rather than revert. Test updated to reflect new behavior.
- **Commit**: `2808d8c`

### 🟡 CCM-1 (LOW): `setFeeAddress` missing zero-address check
- **Source**: ClientManager+aux audit agent
- **File**: `CawClientManager.sol::setFeeAddress`
- **Bug**: `createClient` enforces non-zero `feeAddress` but
  `setFeeAddress` didn't. A client owner could zero out the fee address,
  causing payFee to accrue to address(0) and `withdrawFees` to revert
  forever (CAW.transfer to address(0) reverts on standard ERC-20s).
- **Fix**: `require(feeAddress != address(0))` mirroring `createClient`.
- **Commit**: `7e3b2e4`

## Round 2 findings deferred (NOT fixed in this commit set)

The following are real issues but require design discussion or have
operational complexity beyond a single-line fix. Listed in priority
order for follow-up work.

### 🔴 Archive HIGH-1: Future-checkpoint attack
- **Source**: Archive audit agent
- **Bug**: A validator can submit `submitReplication` for checkpoints
  that haven't been written on the source L2 yet. If L2 doesn't catch
  up within the 2-day challenge window, neither Mode A
  (`slashIncoherentRoot`) nor Mode B (`relayChallenge`) can fire (the
  source L2's `clientHashAtCheckpoint(cp) == 0`, so the relay reverts
  with "Checkpoint does not exist"). The fraudulent submission
  finalizes; checkpoint claims become permanent; honest validators can
  never re-replicate those cps.
- **Fix needs**: an L2-liveness oracle/proof mechanism (e.g., periodic
  LZ heartbeat from the relay carrying the latest cp seen on L2).
  Architectural change.

### 🔴 Archive HIGH-2: No `srcEid` binding (multi-source-chain unsafe)
- **Source**: Archive audit agent
- **Bug**: A `Submission` has a `clientId` but no `srcEid`. `clientId`s
  are scoped per source chain (Base's clientId=5 ≠ Optimism's clientId=5).
  Once a second source chain is peered, an attacker on the second chain
  can deliver a "fraud proof" (with that chain's unrelated clientId=5
  hash) and slash an HONEST validator on the first chain.
- **Fix needs**: add `srcEid` to `Submission`; bind via per-clientId
  registry populated by an LZ "claim" message from the relay; check
  `_origin.srcEid == sub.srcEid` in `_processChallenge`.
- **Status**: latent — only activates when 2nd source chain is peered.

### 🟠 L1 M-1: Silent ETH retention on bypassLZ / empty-queue paths
- **Source**: CawProfile L1 audit agent
- **Bug**: Many call sites compute `lzEthAmount = msg.value - fees` and
  pass to `_updateNewOwners`. If no LZ send fires (bypassLZ branch or
  empty queue), the `lzEthAmount` is silently absorbed by the contract
  with no refund.
- **Fix needs**: refund excess ETH at every call site, or reject
  overpayment in bypassLZ branches.

### 🟠 L2 M-1: LZ unordered delivery → ownership desync window
- **Source**: CawProfileL2 audit agent
- **Bug**: Token transferred Alice→Bob, then Bob→Carol on L1. If LZ
  delivers Bob→Carol message before Alice→Bob, L2 settles to Bob (incorrect),
  and Bob's still-valid session keys can drain N's L2 balance during
  the desync window.
- **Fix needs**: override `nextNonce` to enforce ordered delivery, OR
  per-token transfer counter in the LZ payload to ignore stale updates.

### 🟠 L2 M-3: registerSessionPersonal replayable across L2 deployments
- **Source**: CawProfileL2 audit agent
- **Bug**: The personal-sign message format includes spend limit, tip
  rate, expiry, session key — but NO chain ID, NO contract address.
  Same signature replays on every L2 the contract is deployed to.
- **Fix needs**: include chain id + contract address in the
  human-readable message format (FE coordinated change).

### 🟠 L2 M-4: `_parseExpiryValue` accepts invalid date components
- **Source**: CawProfileL2 audit agent
- **Bug**: Day check is `1 <= day <= 31` (not month-aware). "Feb 31" or
  "30:99:99 hours" parse and silently roll over to a different date.
- **Fix needs**: month-aware day validation, range checks for hour/minute/second.

### 🟠 CawActions M-2: ZK path session-revoke front-run reverts whole batch
- **Source**: CawActions audit agent
- **Bug**: `_zkProcessOneGroup` line 427's expiry check
  `require(expiry > block.timestamp)` reverts the whole batch on a
  session revoked between proof generation and submission. Breaks the
  ZK path's "skip-don't-revert" promise.
- **Fix needs**: skip the affected group (set bits to 0 in
  `actionsExecutedBitmap`) rather than revert. Design decision.

### 🟠 Archive MED-3: `setDelegate` not OnlyOnce-locked
- **Source**: Archive audit agent
- **Bug**: `OAppCore.setDelegate(address)` is `onlyOwner` and not
  overridden with `OnlyOnce`. Pre-renouncement, a compromised owner key
  could set a malicious delegate.
- **Fix needs**: lock `setDelegate` post-deploy.

### 🟠 Archive MED-1: `_payNative` traps overpaid native fees in relay
- **Source**: Archive audit agent
- **Bug**: `CawChallengeRelay._payNative` returns `_nativeFee` (not
  `msg.value`), so any overpayment stays in the relay forever (no
  withdraw path).
- **Fix needs**: return `msg.value` so the LZ endpoint refunds excess
  to the caller.

### Other Marketplace findings
- **M-1 (MEDIUM)**: blocklist-prone tokens (USDC) can trap bid refunds
  if the bidder is blocklisted post-bid. Pull-pattern `withdrawBidTo(recipient)`

- **L-4**: no upper bound on offer `duration`.
- **L-6**: marketplace allows seller to buy own listing (wash trade).

## What changed about my recommendation

The first audit pass found 4 bugs. Round 2 found **9 more** including 3
HIGH-severity issues. That's a real argument that more passes find more
bugs — and that I shouldn't have stopped after the first round.

For mainnet deployment: don't rely on this pass alone. The pattern of
finding HIGH-severity bugs in every fresh-eyes pass strongly suggests
there are more I haven't found. Plan more review passes — ideally with
fuzzing and/or symbolic execution. The bugs that remain undiscovered
are the dangerous ones.
