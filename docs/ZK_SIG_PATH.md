# ZK Sig-Only Path

Operator-facing guide to `processActionsWithZkSigs` ‚Äî the second entry point
into CawActions, where signature recovery is moved off-chain into a Groth16
proof generated by an SP1 zkVM circuit.

For the full sig path (ecrecover-on-chain), see
`solidity/contracts/CawActions.sol::processActions`. This doc covers what's
different about the ZK path, when to use it, what it costs, and how to run it.

---

## TL;DR

- The ZK path replaces per-action `ecrecover` with one constant-cost
  `verifyProof` call (~**265K gas** on Base Sepolia, measured directly
  against the canonical bytecode on a fork).
- **Break-even vs the sig path: ~n ‚â• 70 actions per batch** (singleton sig
  groups, mixed-signer batches ‚Äî what real prod batches look like). Below
  that, sig path is cheaper; above that, ZK path is cheaper.
- **Real prod batches today (n=20‚Äì30) are MORE expensive on the ZK path
  by 167K‚Äì209K gas (~+25%).** ZK is not a savings at current batch sizes;
  it only pays off if a validator can sustainably batch ‚â•70 actions per tx.
- Cawonce conflicts SKIP the affected slots (don't revert the batch). The
  sig path keeps the all-or-nothing semantic.
- The path is optional. An install with `zkVerifier == address(0)` simply
  reverts every call to `processActionsWithZkSigs`; nothing else changes.
- **Local proving needs ~16 GB RAM**. The 5.9 GB test.caw.social VPS cannot
  run it; hosted prover networks (SP1's, ~10s/proof) are the practical path
  for low-RAM hosts.

## When to use which path

| Situation | Path | Why |
|-----------|------|-----|
| Small batches (‚â§ 69 actions) | sig | Verifier's fixed cost dominates the ecrecover savings. |
| Large batches (‚â• 70 actions) | ZK | Per-action savings beat the verifier's fixed cost. |
| Race-prone slots (cawonce already used) | ZK | Skip-don't-revert keeps the rest of the batch running. |
| Simulation / estimateGas | sig | Existing tooling assumes all-or-nothing semantics. |
| ZK prover unavailable | sig | Always works, never blocks on a worker. |

`ValidatorService` picks transparently based on whether a proof has been
staged in `zkProofCache` for the exact `(packedActions, packedSigs)` tuple
and whether `ZK_PROVER_ENABLED=1` is set. Cache miss = sig path. No proof
worker yet = always sig path. See "Validator integration" below.

## What the proof attests

The Groth16 proof commits to four `bytes32` public values:

```
public_input[0] = keccak256(packedActions)     // the wire bytes the verifier sees
public_input[1] = keccak256(packedSigs)        // grouped (v, r, s) tuples
public_input[2] = keccak256(signers)           // 20-byte addresses, one per action
public_input[3] = eip712DomainHash             // pinned to the deployed CawActions
```

It does **not** commit to chain state ‚Äî no cawonces, no balances, no hash
chain. That's the property that makes the path race-safe: a competing tx
between proof generation and submission can't invalidate the proof, because
the proof never claimed anything about chain state. The contract still does
all the state checks; the proof just lets it skip the cryptographic part.

The circuit lives at `solidity/zk/sig-recovery/program/src/main.rs`. It walks
the same group structure the contract walks, recomputes the EIP-712 digest
for each action, runs ECDSA recovery via SP1's audited `k256` precompile, and
asserts the recovered address matches the corresponding 20-byte slot in the
`signers` blob.

## Skip-don't-revert: the practical reason it exists

Without ZK, a single race-loss aborts the whole batch and the validator pays
gas for an empty receipt. With ZK, the batch keeps running and the
`actionsExecutedBitmap` field of the `ActionsProcessedZk` event tells the
indexer which slots actually ran:

```
event ActionsProcessedZk(bytes packedActions, uint256 actionsExecutedBitmap);
```

Bit `i` set = action `i` executed. Bit `i` clear = action `i` was skipped
(its cawonce was already consumed by another tx). The validator service
filters these out of its downstream pipeline so the unexecuted actions get
retried via the normal sig-path lifecycle.

This matters because the validator submits proofs that take 10s+ to generate;
during that window, another sig-path tx may consume the same cawonce. We'd
rather lose one slot than the whole batch.

## Known issues found and fixed during review

A systematic attack-surface review on 2026-05-07 turned up two bugs in the
skip-don't-revert path. Both are fixed in commit `5e2bc20`. Documented here
for the audit trail.

  1. **Skipped WITHDRAW double-credit on L1 (security).** `_trackClientAndWithdraw`
     was called BEFORE the cawonce-skip check, so a skipped WITHDRAW slot
     still set its bit in `withdrawBitmap`. The trailing `_handleWithdrawals`
     pass would then queue a `setWithdrawable` LZ message for an action that
     never actually debited L2. Result: with a competing sig-path tx
     consuming the same cawonce first, the user got their withdraw amount
     credited on L1 *twice* for one L2 debit. Fix: defer the bitmap update
     to after the skip check. Regression test:
     `solidity/test/zk-actions-test.js` "regression: skipped WITHDRAW does
     not double-credit on L1".
  2. **`implicitTipOwed` never accumulated (validator economics).** The ZK
     path called `_applyAction(...)` and discarded its return value, while
     the sig path's `_processSingleSig` writes
     `c.implicitTipOwed += _applyAction(...)`. Validators running the ZK
     path on session-key-signed batches were silently never credited the
     per-action tips. Not exploitable by users ‚Äî just a financial bug
     against the validator. Fix: capture the return value, same as the
     sig path. (Caught by code reading, no exploit-style test added because
     the fix is a single character `+=` matching the sig path's pattern.)

There may be other bugs we haven't found yet. The path is **not audited**.
The on-chain state-application code is small (~150 lines under the
`processActionsWithZkSigs` entry point) and mirrors `processActions` for
its non-ZK parts; an audit before mainnet ZK deployment is recommended.

## Trust model

The ZK path **adds** to the verification surface; it doesn't replace any
checks the sig path does.

| Layer | sig path | ZK path |
|-------|----------|---------|
| Cawonce uniqueness | yes (revert on conflict) | yes (skip on conflict) |
| Hash chain advance | yes | yes |
| EIP-712 domain | yes (recomputed) | yes (committed in proof) |
| Sig recovery | on-chain ecrecover | off-chain SP1 circuit |
| Verifier code | EVM precompile | SP1Verifier (`0x397A5f7f...`) |

The ZK path's additional trust assumptions, in plain English:

1. **The Groth16 verifier is sound** for the specific verifier version we
   call. Succinct's `SP1VerifierGateway` routes by selector; we use a
   registered version. Vendored copy in `solidity/contracts/sp1-vendor/` for
   local tests; on prod we point at the canonical address.
2. **The circuit is correct.** Equivalence-tested byte-for-byte against
   `_computeStructHash` in Solidity (`solidity/test/zk-digest-equivalence-test.js`)
   so any drift between Rust and Solidity surfaces in CI.
3. **The vkey pinned in CawActions matches the circuit binary** the prover
   runs. Mismatch = every proof rejects. Pinned at deploy time, immutable.

The signers blob arrives in plain calldata, but the proof's `signersHash`
commitment prevents a malicious validator from substituting addresses ‚Äî the
20-byte slot at position `i` MUST be the address SP1's circuit recovered for
action `i`.

## Validator integration (today)

`ValidatorService/index.ts` consults two things on every batch submit:

1. `process.env.ZK_PROVER_ENABLED === '1'` ‚Äî gate.
2. `zkProofCache.get(keccak256(packedActions || packedSigs))` ‚Äî does a proof
   for THIS exact tuple already exist?

If both are true, calldata is encoded for `processActionsWithZkSigs` and the
proof is consumed (single-use ‚Äî retries fall back to sig automatically). If
either is false, sig path. There's no difference in observable behavior from
the validator's perspective other than the function selector and the event
shape.

The `zkProofCache` map is populated by `stageZkProof()`, exported from the
same module. Today, no internal worker calls `stageZkProof()` ‚Äî we deliberately
left the producer dormant until the queueing shape (per-mirror?
proof-every-batch vs every-Nth?) is decided. Until then, **`ZK_PROVER_ENABLED=1`
is harmless**: the cache stays empty and behavior is identical to today.

To drive the path manually, run the standalone CLI:
```bash
cd client
ZK_PROVER_ENABLED=1 npx tsx scripts/zk-prove-and-submit.ts <multiData.json>
```
where `multiData.json` is a captured batch (see the script's header for the
JSON shape). The script proves locally, sanity-checks the signers' keccak,
and submits via `processActionsWithZkSigs` directly.

## Resource cost of proving

| Step | Cost (this Mac) | Cost (hosted) | Notes |
|------|-----------------|---------------|-------|
| First-time SRS download | ~90 min, 5.79 GB | n/a | One-shot. Subsequent proofs reuse the extracted cache. |
| Build the circuit ELF | ~1 min | n/a | `cargo build --release`. Fixed per circuit revision. |
| Witness gen + prove core | ~20s | <1s | CPU-bound. |
| Wrap to Groth16 | ~14s, **~16 GB peak RAM** | <1s | The expensive step. |
| Total per proof (warm) | **~34s** | **~10s** | After everything else is cached. |

The 16 GB peak RAM during wrap is the headline number. On a 5.9 GB VPS with
no swap, this WILL OOM-kill the host (likely Postgres or pm2 itself). For
low-RAM hosts the only sane option is the hosted prover network (SP1's:
`SP1_PROVER=network` + `NETWORK_PRIVATE_KEY`, returns proofs in ~10s).

## What can change later (and what can't)

**Can change** (without redeploying CawActions):
- The validator's queueing strategy
- Whether to use Mac-local or hosted proving
- The proof-cache eviction policy
- Frontend UX for ZK-path submissions

**Cannot change** (immutable in CawActions):
- `zkVerifier` address ‚Äî pinned to a specific gateway/verifier
- `zkProgramVKey` ‚Äî pinned to the circuit binary's vkey hash
- `eip712DomainHash` ‚Äî pinned to (chainId, address)

If the SP1 circuit changes (different program ‚Üí different vkey), or Succinct
ships a new verifier version that we want to migrate to, we deploy a new
CawActions and migrate. There is no upgrade path in-place ‚Äî by design, since
the verification trust root has to be tamper-evident.
